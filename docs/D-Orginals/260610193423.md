# 一文讲懂 Skill 和 Agent 的区别与联系

在学习 Hermes Agent、OpenManus、LangGraph、AutoGPT 这类 AI Agent 框架时，很多人都会遇到一个很容易混淆的问题：

**Skill 和 Agent 到底有什么区别？**

更让人困惑的是，Skill 里面也可以设计一套前后衔接的流程，Agent 也可以调用工具、拆解任务、连续执行。于是很多人会产生一种感觉：

**Skill 好像也是 Agent，Agent 好像也能做 Skill 的事。**

这种感觉并不是错觉，因为它们确实有重叠的地方。但从工程设计角度看，Skill 和 Agent 并不是同一个层级的东西。

简单来说：

**Agent 是“大脑”，负责判断和决策。**

**Skill 是“专家能力”，负责把某类任务稳定地完成。**

如果用一个更容易理解的比喻：

```text
Agent 像项目经理或 CEO
Skill 像某个专业部门
Tool 像具体工具
```

比如一个公司里，CEO 不会亲自做财务报表、写合同、做海报、发邮件。他更重要的作用是判断现在要做什么、先做什么、交给谁做、做完后是否合格。

而财务部、法务部、市场部、技术部，则分别负责把某一类专业任务做好。

放到 AI 系统中也是类似的：

```text
Agent
  ↓
Skill
  ↓
Tool
```

Agent 负责调度 Skill，Skill 负责组织流程，Tool 负责执行具体动作。

---

## 一、Skill 是什么？

在 Hermes 这类系统中，经常会看到这样一句话：

**Skill = SOP + 工具 + 知识库**

这句话很关键。

所谓 Skill，本质上就是一个被封装好的业务能力。它不是单纯的一条提示词，也不是单纯的一个工具，而是把一类任务的执行方法固定下来，形成一个可复用的能力模块。

比如：

```text
写周报 Skill
```

它可能包含：

```text
输入本周工作内容
↓
整理重点事项
↓
归类项目进展
↓
提炼成果
↓
补充问题与计划
↓
输出标准周报
```

再比如：

```text
AI 日报 Skill
```

它可能包含：

```text
搜索 AI 新闻
↓
筛选重要内容
↓
总结 10 条资讯
↓
生成日报格式
↓
推送到飞书群
```

所以，Skill 不是只做一个动作。它可以包含多个步骤，也可以调用工具，也可以读取知识库，还可以按照固定格式输出结果。

这就是为什么 Skill 看起来也能“前后衔接”。

因为 Skill 内部本来就可以有流程。

但是，这个流程一般是预先设计好的。

Skill 的核心特点是：

```text
目标固定
流程固定
输入输出固定
适合重复执行
```

换句话说，Skill 更像是一套标准作业流程，也就是 SOP。

它关心的是：

```text
这类任务应该怎么稳定完成？
```

---

## 二、Agent 是什么？

Agent 的核心不是“流程”，而是“决策”。

Agent 更像一个能够根据目标动态思考、拆解任务、选择工具、调用 Skill、检查结果的智能体。

比如用户说：

```text
帮我分析一下最近 AI Agent 的发展趋势，生成一份 PPT，然后发到飞书群。
```

这个任务不是一个单一动作，而是一个复杂目标。

Agent 可能会自动拆解为：

```text
第一步：搜索最近 AI Agent 相关新闻
第二步：筛选有价值的信息
第三步：总结发展趋势
第四步：生成 PPT 大纲
第五步：制作 PPT 文件
第六步：上传或发送到飞书群
第七步：检查任务是否完成
```

在这个过程中，Agent 要不断判断：

```text
现在应该先做什么？
需要调用哪个工具？
是否需要搜索？
是否需要生成文件？
结果是否合格？
如果失败了怎么办？
下一步应该继续还是终止？
```

这才是 Agent 的核心价值。

所以，Agent 关心的是：

```text
面对一个目标，我应该如何规划并完成它？
```

Agent 的特点是：

```text
目标可能变化
路径可能变化
工具选择可能变化
执行顺序可能变化
中间结果可能影响下一步
```

这就是 Agent 和 Skill 最大的区别。

---

## 三、为什么 Skill 也可以做前后衔接？

因为 Skill 本身可以封装流程。

很多人误以为 Skill 只是一个“单点工具”，比如翻译、搜索、发邮件。其实不是。

一个成熟的 Skill 可以包含很多连续步骤。

比如“市场调研 Skill”可能包含：

```text
确定调研主题
↓
搜索公开资料
↓
提取核心观点
↓
整理竞品信息
↓
归纳市场趋势
↓
输出调研报告
```

这当然是前后衔接的。

但是要注意，Skill 的衔接通常是提前设计好的。

也就是说，它一般遵循固定路线：

```text
永远是 A → B → C → D
```

它不会每次都重新思考：

```text
这次我要不要先做 B？
这次要不要跳过 C？
这次要不要调用另一个 Skill？
这次任务失败后要不要换策略？
```

如果它能大规模动态判断，那它就越来越接近 Agent 了。

所以可以这样理解：

**Skill 可以有流程，但流程通常是固定的。**

**Agent 也可以有流程，但流程通常是动态生成的。**

这是二者最关键的差异。

---

## 四、为什么 Agent 也能做 Skill 的事？

因为 Agent 本身具备调用工具、组织步骤、完成任务的能力。

比如用户说：

```text
把这篇文章翻译成英文。
```

Agent 完全可以自己完成：

```text
读取文章
↓
理解内容
↓
翻译
↓
输出结果
```

它不一定非要调用一个“翻译 Skill”。

所以从能力上说：

**Agent 可以直接完成很多 Skill 能做的事。**

但是从工程实践上看，不建议什么事情都让 Agent 自由发挥。

原因很简单：

```text
不稳定
不可控
难复现
成本高
容易跑偏
```

比如你每天都要生成一份 AI 日报。如果完全交给 Agent 自由发挥，可能今天它搜索 10 条，明天搜索 20 条；今天格式很规范，明天格式又变了；今天能成功发到飞书，明天可能忘了推送。

这对个人尝鲜也许可以接受，但对企业级应用来说是不合格的。

企业需要的是：

```text
稳定
可控
可复现
可维护
```

所以会把高频、固定、重要的任务封装成 Skill。

这样每次执行都尽量走同一套标准流程。

## 五、Skill 和 Agent 的本质区别

可以从几个维度来看。

### 1. 角色不同

Agent 是决策者。

Skill 是执行能力。

Agent 负责判断“做什么”和“怎么安排”。

Skill 负责解决“这类事情具体怎么做”。

### 2. 灵活性不同

Agent 更灵活，可以根据任务变化调整路径。

Skill 更稳定，按照预设流程执行。

### 3. 适用场景不同

Agent 适合开放式、复杂、多步骤、不确定的任务。

Skill 适合重复性强、流程明确、标准化程度高的任务。

### 4. 工程价值不同

Agent 的价值是智能调度。

Skill 的价值是稳定复用。

一个系统如果只有 Agent，可能很聪明，但不稳定。

一个系统如果只有 Skill，可能很稳定，但不够灵活。

真正好用的 AI 系统，往往是：

```text
Agent 负责思考和调度
Skill 负责稳定执行
Tool 负责具体操作
```

## 六、用一个例子彻底讲清楚

假设用户提出一个任务：

```text
帮我整理最近 AI Agent 的发展趋势，做成 PPT，并发到飞书群。
```

如果系统里有 Agent，它会先拆解任务：

```text
1. 需要搜索 AI Agent 最新资讯
2. 需要筛选重点内容
3. 需要总结趋势
4. 需要生成 PPT
5. 需要发送到飞书群
```

然后 Agent 可能会调用多个 Skill：

```text
资讯搜索 Skill
趋势分析 Skill
PPT 生成 Skill
飞书推送 Skill
```

每个 Skill 内部又有自己的 SOP。

比如“资讯搜索 Skill”负责：

```text
搜索关键词
筛选来源
去重
提取摘要
返回结构化信息
```

“PPT 生成 Skill”负责：

```text
生成标题
拆分章节
制作页面结构
生成 PPT 文件
```

“飞书推送 Skill”负责：

```text
检查群机器人配置
上传文件
发送消息
返回发送结果
```

这时候，整个系统的层级就很清楚了：

```text
Agent：决定整个任务怎么完成
Skill：负责完成某个专业子任务
Tool：执行最底层的具体动作
```

## 七、为什么 Hermes 这类系统强调 Skill？

因为 Hermes 这类系统并不是只追求“让 AI 自由发挥”。

它更强调：

```text
把人的经验沉淀成可复用能力
```

也就是把复杂任务中反复出现的流程、规则、工具、知识库、注意事项封装成 Skill。

这样做的好处是：

```text
第一次做任务，靠 Agent 探索
第二次做任务，总结成经验
第三次做任务，封装成 Skill
以后重复做任务，直接稳定复用
```

这就是从“聊天式 AI”走向“工程化 AI 助理”的关键。

普通聊天模型更多是临场回答。

Agent 可以临场规划和执行。

Skill 则把成功经验沉淀下来，让系统越来越稳定。

所以 Skill 的意义不是取代 Agent，而是让 Agent 更可靠。

---

## 八、最容易混淆的一点：复杂 Skill 和小 Agent 很像

现实中，Skill 和 Agent 的边界并不是绝对清晰的。

一个简单 Skill 可能只是：

```text
翻译文本
```

一个复杂 Skill 可能是：

```text
搜索资料
↓
分析内容
↓
生成报告
↓
保存文件
```

这个复杂 Skill 看起来确实像一个“小 Agent”。

但判断它到底更像 Skill 还是 Agent，可以看一个关键问题：

**它是在固定流程中执行，还是在动态决策中执行？**

如果它主要是按照预设步骤走，那它就是 Skill。

如果它会根据目标不断判断下一步怎么做，那它就更像 Agent。

所以：

```text
固定流程 = 更像 Skill
动态决策 = 更像 Agent
```

---

## 九、最终总结

Skill 和 Agent 的关系，可以用一句话概括：

**Agent 负责“想清楚要怎么完成目标”，Skill 负责“把某类任务按照标准流程做好”。**

再简单一点：

```text
Agent 负责做什么
Skill 负责怎么做
Tool 负责实际操作
```

更完整地说：

```text
Agent = 决策与调度中心
Skill = 可复用的专业能力
Tool = 最底层的执行工具
```

Skill 之所以也能前后衔接，是因为它内部可以封装 SOP。

Agent 之所以也能做 Skill 的事，是因为它本身具备执行和组织步骤的能力。

但在真正的工程系统里，二者最好不要混用。

Agent 适合处理不确定性。

Skill 适合沉淀确定性。

一个成熟的 AI Agent 系统，往往不是让 Agent 什么都自由发挥，而是让 Agent 在合适的时候调用已经封装好的 Skill。

这样既能保留智能体的灵活性，又能保证任务执行的稳定性。

这也是 Hermes 这类系统强调 Skill 的根本原因。

它不是单纯让 AI 更会聊天，而是让 AI 逐渐变成一个可复用、可维护、可长期协作的个人或企业级智能助理。
